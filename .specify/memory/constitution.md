# 프로젝트 Constitution

## Core Principles

### I. 스펙 우선
모든 기능은 구현 전에 스펙 문서로 정의하고, 스펙에 명시되지 않은 기능은 구현하지 않는다.
판정 기준: 구현 시작 전 스펙이 존재하고, 변경된 기능이 스펙에 반영되었으며, 구현 결과가 스펙 범위를 벗어나지 않는지 확인할 수 있어야 한다.

### II. 테스트 필수
핵심 기능(추가/완료 토글, 삭제, 남은 개수, 필터)은 pytest 자동화 테스트로 검증한다.
구현 완료의 기준은 해당 테스트가 통과하는 것이다.
판정 기준: 핵심 기능에 대한 테스트가 존재하고, CI나 로컬 실행으로 통과하는지 확인할 수 있어야 한다.

### III. 설정 분리
데이터베이스 경로 등 환경설정 값은 .env 파일로 관리하고 저장소에 커밋하지 않는다.
필요한 환경변수는 .env.example 파일에 문서화한다.
판정 기준: 민감하거나 환경별로 달라지는 값이 코드에 하드코딩되지 않았고, .env.example에 필요한 변수명이 모두 기록되어 있는지 확인할 수 있어야 한다.

### IV. 추적 가능한 커밋
모든 커밋 메시지는 태스크 ID를 앞에 붙여 작성한다. 예: "T03: 할 일 생성 API 구현".
판정 기준: 커밋 메시지 형식이 일관되게 적용되고, 각 변경이 추적 가능한 식별자를 갖는지 확인할 수 있어야 한다.

### V. 단순성
요구사항에 없는 기능이나 라이브러리를 임의로 추가하지 않는다.
판정 기준: 새로운 기능이나 의존성 추가가 요구사항과 직접적으로 연결되어 있고, 더 단순한 대안이 없는지 설명할 수 있어야 한다.

## 추가 제약

이 프로젝트에서 생성되는 모든 문서(constitution, spec, plan, tasks, checklist, 분석 리포트 등)는 반드시 한국어로 작성한다.
이 규칙은 요구사항, 구현 계획, 테스트 결과, 리뷰 의견, 분석 리포트 등 모든 산출물에 적용된다.
판정 기준: 생성된 문서의 제목, 본문, 표, 체크리스트가 한국어로 작성되어 있고, 외국어가 필요할 때도 설명이 한국어로 병기되는지 확인할 수 있어야 한다.

## 개발 워크플로우

모든 구현은 먼저 스펙과 계획을 정리한 뒤 테스트를 추가하고, 그다음 구현을 진행한다.
핵심 기능이 변경되면 관련 테스트를 먼저 갱신하고, 구현 완료 전에는 반드시 테스트를 재실행한다.
환경설정은 .env와 .env.example를 기준으로 관리하고, 커밋·리뷰 전에는 문서와 커밋 메시지 형식이 규칙을 준수하는지 확인한다.
판정 기준: 작업 시작 전 스펙/계획이 준비되어 있고, 변경 후 테스트와 문서 기준이 모두 충족되는지 리뷰 단계에서 확인할 수 있어야 한다.

## Governance

이 Constitution은 프로젝트의 개발 방식과 품질 기준을 우선한다. 모든 구현, 리뷰, 배포 결정은 이 문서의 원칙보다 낮은 우선순위를 가질 수 없다.
개정은 변경 사유, 영향 범위, 검증 결과를 포함한 문서로 작성하고, 관련된 spec/plan/tasks/checklist/분석 리포트까지 함께 갱신해야 한다.
버전은 MAJOR.MINOR.PATCH로 관리한다. 원칙 삭제 또는 재정의는 MAJOR, 새 원칙 추가 또는 기존 원칙의 중요 규칙 확장은 MINOR, 문구 정정·용어 보완 등 비의미적 변경은 PATCH로 올린다.
모든 변경사항은 리뷰어가 준수 여부를 확인한 뒤 반영한다. 리뷰 단계에서 원칙 위반이 발견되면 수정이 완료될 때까지 Merge나 배포를 진행할 수 없다.
