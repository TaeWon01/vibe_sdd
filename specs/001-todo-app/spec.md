# 기능 명세서: 오늘의 할 일 웹 앱

**기능 브랜치**: `001-todo-app`

**작성일**: 2026-07-15

**상태**: Draft

**입력**: 사용자 설명: "오늘의 할 일" 웹 앱을 만들려고 해. 로그인 없이 단일 사용자를 위한 웹 앱으로, 사용자 관점의 요구사항은 다음과 같아.

## 사용자 시나리오 및 테스트

### 사용자 이야기 1 - 새 할 일을 추가한다 (우선순위: P1)
사용자는 할 일 입력창에 내용을 입력하고 추가 버튼을 눌러 목록에 새 항목을 등록한다.

**이 우선순위의 이유**: 할 일 생성은 이 앱의 핵심 진입 기능이며, 이후 완료/삭제/필터링 기능이 모두 이 기반 위에서 동작한다.

**독립 테스트**: 입력값을 넣고 추가하면 목록에 새 항목이 나타나고, 미완료 항목 수만 기준으로 남은 할 일 수가 즉시 갱신되는지 확인할 수 있다.

**수락 시나리오**:
1. 사용자가 빈 값으로 추가를 시도하면, 항목이 추가되지 않고 안내 메시지가 표시된다.
2. 사용자가 공백만 있는 값을 입력하면, 항목이 추가되지 않고 안내 메시지가 표시된다.
3. 사용자가 할 일 제목을 100자를 초과해 입력하면, 항목이 추가되지 않고 안내 메시지가 표시된다.
4. 사용자가 유효한 할 일을 입력하면, 목록에 새 항목이 추가되고 남은 개수가 갱신된다.
5. 새로 추가된 미완료 항목은 목록 상단에 표시되고, 완료된 항목은 목록 하단에 표시된다.

---

### 사용자 이야기 2 - 할 일을 완료하고 다시 되돌린다 (우선순위: P1)
사용자는 목록에서 할 일을 완료 처리할 수 있고, 완료된 항목은 시각적으로 구분된다. 완료된 항목은 다시 미완료 상태로 되돌릴 수 있다.

**이 우선순위의 이유**: 완료 상태 전환은 앱의 핵심 상호작용이며, 상태 변화가 사용자에게 즉시 반영되어야 한다.

**독립 테스트**: 항목을 완료 처리하면 완료 상태로 표시되고, 다시 클릭하면 미완료 상태로 복구되는지 확인할 수 있다.

**수락 시나리오**:
1. 사용자가 미완료 항목을 완료 처리하면, 항목이 완료 상태로 표시되고 미완료 항목 수 기준의 남은 개수가 감소한다.
2. 사용자가 완료된 항목을 다시 클릭하면, 항목이 미완료 상태로 복구되고 미완료 항목 수 기준의 남은 개수가 증가한다.
3. 완료된 항목은 취소선과 흐린 색으로 표시된다.

---

### 사용자 이야기 3 - 할 일을 삭제한다 (우선순위: P1)
사용자는 할 일을 삭제할 수 있으며, 삭제 전 확인 절차를 거친다.

**이 우선순위의 이유**: 삭제는 데이터 변경의 중요한 동작이므로 실수 방지 절차가 필요하다.

**독립 테스트**: 삭제 버튼을 누르면 확인 절차가 표시되고, 확인 후 항목이 제거되는지 확인할 수 있다.

**수락 시나리오**:
1. 사용자가 삭제 버튼을 누르면 브라우저의 확인창(confirm) 수준의 확인 절차가 시작된다.
2. 사용자가 확인을 취소하면 항목은 유지된다.
3. 사용자가 확인을 하면 항목이 목록에서 제거되고 미완료 항목 수 기준의 남은 개수가 갱신된다.

---

### 사용자 이야기 4 - 목록을 필터링하고 상태를 확인한다 (우선순위: P2)
사용자는 전체, 진행중, 완료 3가지 필터로 항목을 구분해 보고, 화면 상단의 남은 할 일 개수를 확인한다.

**이 우선순위의 이유**: 필터와 상태 요약은 사용자가 목록을 빠르게 파악하는 데 필수적인 기능이다.

**독립 테스트**: 필터를 변경하면 해당 조건의 항목만 보이고, 남은 할 일 수가 항상 정확하게 반영되는지 확인할 수 있다.

**수락 시나리오**:
1. 사용자가 "진행중" 필터를 선택하면 미완료 항목만 표시된다.
2. 사용자가 "완료" 필터를 선택하면 완료 항목만 표시된다.
3. 화면 상단의 "남은 할 일 N개"는 추가, 완료, 삭제 동작 후 즉시 갱신되며, 미완료 항목 수만 집계한다.

---

### 사용자 이야기 5 - 앱을 재방문해도 데이터를 유지한다 (우선순위: P2)
사용자는 브라우저를 새로고침하거나 서버를 재시작해도 기존 할 일 목록을 계속 확인한다.

**이 우선순위의 이유**: 데이터 영속성은 사용자가 기대하는 기본적인 앱 동작이며, 신뢰성을 결정하는 핵심 요소다.

**독립 테스트**: 새로고침 또는 서버 재시작 후에도 저장된 할 일 목록이 유지되는지 확인할 수 있다.

**수락 시나리오**:
1. 사용자가 할 일을 추가하거나 수정한 뒤 새로고침하면 데이터가 유지된다.
2. 서버를 재시작한 뒤에도 기존 할 일 목록이 그대로 보인다.

---

### 사용자 이야기 6 - 모바일과 빈 상태를 지원한다 (우선순위: P2)
사용자는 모바일 화면에서도 앱을 사용하고, 할 일이 없을 때 친절한 안내를 받는다.

**이 우선순위의 이유**: 반응형 동작과 빈 상태 안내는 접근성과 사용 편의성을 높이는 핵심 경험이다.

**독립 테스트**: 360px 폭 화면에서 레이아웃이 정상적으로 표시되고, 할 일이 없을 때 안내 문구가 보이는지 확인할 수 있다.

**수락 시나리오**:
1. 화면 폭이 360px인 모바일 환경에서도 주요 기능이 정상적으로 접근 가능하다.
2. 할 일이 하나도 없을 때 빈 상태 안내 문구가 표시된다.

---

### 엣지 케이스

- 입력값이 비어 있거나 공백만 있을 때는 항목이 추가되지 않는다.
- 삭제 확인을 취소한 경우 기존 항목은 유지된다.
- 필터가 변경되어 현재 보이는 항목이 없더라도 빈 상태 안내가 적절히 표시된다.
- 저장된 데이터가 비어 있을 때도 앱은 정상적으로 동작한다.

## 요구사항

### 기능 요구사항

- **FR-001**: 시스템은 사용자가 새 할 일을 입력하고 추가할 수 있는 기능을 제공해야 한다.
- **FR-002**: 시스템은 빈 값 또는 공백만 있는 입력에 대해 항목 추가를 거부하고, 사용자에게 안내 메시지를 표시해야 한다.
- **FR-003**: 시스템은 할 일 제목의 최대 길이를 100자로 제한하고, 100자를 초과하는 입력에 대해 항목 추가를 거부하고 안내 메시지를 표시해야 한다.
- **FR-004**: 시스템은 할 일을 완료 상태로 전환할 수 있는 기능을 제공해야 한다.
- **FR-005**: 시스템은 완료된 할 일을 다시 미완료 상태로 되돌릴 수 있는 기능을 제공해야 한다.
- **FR-006**: 시스템은 할 일을 삭제할 수 있어야 하며, 삭제 전에는 브라우저 확인창 수준의 확인 절차를 거쳐야 한다.
- **FR-007**: 시스템은 화면 상단에 "남은 할 일 N개"를 표시해야 하며, 할 일 추가/완료/삭제 시 즉시 갱신되어야 하고, 기준은 미완료 항목 수여야 한다.
- **FR-008**: 시스템은 전체, 진행중, 완료 3가지 필터를 제공해야 한다.
- **FR-009**: 시스템은 할 일 데이터를 영구적으로 저장해야 하며, 새로고침이나 서버 재시작 후에도 유지되어야 한다.
- **FR-010**: 시스템은 모바일 화면(최소 360px 폭)에서도 사용할 수 있도록 반응형 레이아웃을 제공해야 한다.
- **FR-011**: 시스템은 할 일이 하나도 없을 때 친절한 빈 상태 안내 문구를 표시해야 한다.
- **FR-012**: 완료된 항목은 시각적으로 구분되어야 하며, 취소선과 흐린 색으로 표시되어야 한다.
- **FR-013**: 목록은 최신 등록된 미완료 항목이 위쪽에, 완료된 항목은 아래쪽에 표시되는 정렬 순서를 가져야 한다.

### 핵심 엔터티

- **할 일(Todo)**: 사용자가 관리하는 작업 항목으로, 내용, 완료 여부, 생성 시점 정보를 포함한다.

## 성공 기준

### 측정 가능한 결과

- **SC-001**: 사용자가 새 할 일을 추가한 후 1분 내에 목록에 반영되고 미완료 항목 수 기준의 남은 개수가 즉시 갱신되어야 한다.
- **SC-002**: 사용자가 완료/삭제/복구 동작을 수행한 후, 상태 변화가 화면에 즉시 반영되어야 하며 남은 개수도 미완료 항목 수 기준으로 갱신되어야 한다.
- **SC-003**: 사용자가 새로고침하거나 서버를 재시작한 뒤에도 저장된 할 일 목록이 유지되어야 한다.
- **SC-004**: 360px 폭 화면에서도 주요 기능에 접근할 수 있어야 한다.
- **SC-005**: 할 일이 없을 때 사용자가 빈 상태 안내를 이해할 수 있어야 한다.
- **SC-006**: 할 일 제목이 100자를 초과하는 경우 사용자에게 명확한 안내 메시지가 표시되어야 한다.
- **SC-007**: 목록에서 미완료 항목은 완료된 항목보다 위에 표시되어야 하며, 새로 등록된 항목은 상단에 노출되어야 한다.

## 가정

- 사용자는 웹 브라우저를 통해 앱을 이용한다.
- 이 앱은 로그인 없는 단일 사용자용으로, 여러 사용자 계정을 구분하지 않는다.
- 할 일 데이터는 로컬 저장소 또는 서버 저장소를 통해 유지되며, 구현 방식은 계획 단계에서 결정한다.
- 모바일 화면 지원은 360px 폭 기준으로 검증한다.
- 삭제 확인 절차는 브라우저 확인창(confirm) 수준의 간단한 방식으로 제공한다.
